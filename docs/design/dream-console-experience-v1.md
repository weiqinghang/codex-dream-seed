# Dream Console 体验基线（V1）

状态：方案 B“聚焦工作台”已确认并进入 0.4.0 开发线原型实现；尚未完成最终体验验收。

## 0. 当前开发状态

当前原型已经实现注意力窗口、改进生命周期和 Console → Codex 的确定性交接，但它形成于
Dream 内核引入 `user_anchor` 之前。因此，这一版只作为可验证的开发基线，不作为最终
Console 方案或稳定发布候选。

下一轮 Console 设计必须补齐：在 Dream 总览和详情中呈现用户体感、原本预期及其证据核对
结果；区分 `aligned`、`partially_aligned`、`conflicting` 和 `insufficient_evidence`；同时
保持 Console 不调用模型、不承担语义复盘的既有边界。

## 1. 产品角色

Dream Console 不是用来浏览内部数据表的管理后台，而是一次 Dream 结束后的行动中心。它帮助用户回答四个问题：

1. 这次 Dream 看见了什么？
2. 哪些结论值得我现在处理？
3. 我愿意让哪些建议进入试用？
4. 后续证据是否支持继续、调整或结束试用？

Console 不替用户决定知识应如何使用，也不在用户不知情的情况下实施候选改进。

Console 是 Codex 的复盘与决策伴侣，不替代 Codex 成为新的工作场所。用户在 Codex 中工作和产生实验数据，只在 Console 中审查复盘结果、作出简单决策，并追踪已决定事项是否最终落实。

## 2. 已确认的产品原则

- Dream 的主要入口仍是 Codex 对话中的“开始做梦”，V1 不在网页重复提供启动入口。
- 首页优先展示“待我处理”和最近一次 Dream，不以累计会话数、任务数等系统统计作为主角。
- 首页“待我处理”在任何时刻最多展示 5 项，候选建议也包含在这 5 项之内。
- 系统负责从完整结果中筛选最关键的少量事项，并解释其优先原因；不把理解和筛选成本转嫁给用户。
- 候选项的首个正向动作叫“进入试用”，不叫“接受”。
- “进入试用”只表达用户意图；真正应用到知识或工作方式前必须再次确认具体变更。
- 系统可以提出简单实验计划、提醒日期和成功标准，但必须由用户确认后才能建立实验。
- 实验固化时必须再次确认影响载体及作用范围，尤其要区分项目级与整个用户环境级。
- 后续证据主要由未来的 Dream 增量积累；网页允许用户补充简短的人工反馈。
- Console 不嵌入模型、不自行进行语义分析或执行项目变更；所有智能工作集中在 Codex。
- 用户确认计划后，Console 创建“等待 Codex 接续”的确定性交接，并明确引导用户回到 Codex。
- Codex 必须先领取交接，再执行授权范围内的工作并回写结果；页面不能把“已决定”显示为“已执行”。
- 主导航调整为：首页、梦境、改进追踪、知识库。
- 任务与证据是 Dream、实验或知识项的上下文详情，不作为一级导航。
- 用户必须能够理解每个状态的含义、下一步动作及其后果。

## 3. 核心用户旅程

```text
在 Codex 中正常工作
  -> 说“开始做梦”
  -> 回答近期项目、环节、体感与原本预期；或明确选择默认复盘
  -> 查看本次扫描范围预览
  -> 用户确认体感锚点、扫描范围和排除项
  -> Dream 增量复盘并持久化结果
  -> 打开 Console 的本次 Dream 摘要并核对体感与证据关系
  -> 理解发现、知识变化和待决策候选
  -> 对候选选择：稍后处理 / 不试用 / 进入试用
  -> 再次确认将采用的具体改进
  -> Console 显示“等待 Codex 接续”并提供接续指令
  -> 回到 Codex，Codex 领取交接并建立改进实验
  -> 回到 Codex 执行真实任务并产生实验数据
  -> 后续 Dream 汇集新的支持证据与反例
  -> 用户决定：继续 / 调整 / 固化 / 结束
  -> Console 追踪已确认的沉淀是否写入并验证完成
  -> 回到 Codex 继续工作
```

### 3.1 关键终态

- 本次无新增内容：明确告诉用户没有需要处理的新发现，并保留扫描范围与时间。
- Dream 已完成、有待处理项：进入首页行动队列和本次 Dream 摘要。
- 候选暂缓：保留理由和提醒语义，不伪装成已完成。
- 候选不试用：记录人的决定及可选原因，允许未来新证据形成新的候选，但不篡改历史决定。
- 候选进入试用：在二次确认前不应用；确认后形成可追踪实验。
- 实验证据不足：保持进行中，明确缺少什么证据。
- 实验结论形成：由用户选择固化、调整或结束。
- 运行失败或结果不完整：显示失败阶段、已保存内容和可恢复动作，不把部分结果展示成完整 Dream。

## 4. 信息架构

| 一级页面 | 用户问题 | 主要内容 | 主要动作 |
| --- | --- | --- | --- |
| 首页 | 我现在最应该处理什么？ | 待处理队列、最近 Dream 摘要、进行中实验、次要系统状态 | 处理候选、查看 Dream、查看实验 |
| 梦境 | 过去每次 Dream 得到了什么？ | Dream 时间线、范围、发现、知识变化、运行完整性 | 打开详情、按状态筛选 |
| 改进追踪 | 所有改进现在分别处于什么阶段，是否真正落实？ | 候选、实验、待落实、已完成与已结束事项的统一列表 | 筛选、打开单项、处理下一步 |
| 知识库 | 当前有哪些可依赖的知识？ | 已验证知识、来源、适用范围、版本与关联实验 | 查看沿革与证据 |

## 5. 首页内容契约

首页不是总览仪表盘，而是按行动优先级组织：

1. **待我处理**：最重要区域。每项说明“为什么现在需要你”“决定会影响什么”“可选动作”。
2. **最近一次 Dream**：时间、范围、完成状态、三条以内关键发现、知识变化和进入完整报告的入口。
3. **进行中的改进实验**：只展示需要关注或最近有新证据的实验；没有变化的实验弱化显示。
4. **系统状态**：扫描游标、工作区健康、最近错误等，仅在异常或用户主动展开时突出。
5. **查看全部改进事项**：进入“改进追踪”，查看不受首页 TOP 5 限制的完整生命周期列表。

首页避免：

- 把原始 ID、文件路径、游标值作为主要内容。
- 用多个大数字制造“数据很多”的感觉。
- 在同一张卡片中并列过多等权动作。
- 用“已接受”暗示改进已经应用。

### 5.1 五项注意力预算

- “待我处理”是一个总量不超过 5 的注意力预算，而不是每种事项各展示 5 项。
- 系统必须保留完整候选池、历史排序和证据；TOP 5 只是本轮最值得用户理解和决定的事项。
- 候选主要由用户在真实任务中的行为和结果反复触发，不靠系统凭空生成。
- 排序同时考虑“近期强度”和“长期累积负担”，避免只追逐最近一两次出现的显眼问题。
- 每项必须用自然语言解释“为什么它进入前五”，不向用户暴露难以解释的综合分数。
- 一项被处理、暂缓或失去优先级后，下一项才能进入首页；历史内容仍可从对应 Dream 或实验中追溯。
- 如果没有足够可信的事项，允许只展示 0–4 项，不为凑满五项生成低价值建议。

#### 综合评定因素

系统可以在内部使用综合评定，但对用户展示构成原因而非黑盒分数：

- **触发频率**：相似行为或结果在多少任务中重复发生。
- **价值影响**：问题对目标质量、完成度或用户收益造成多大影响。
- **绕路成本**：造成多少返工、额外步骤、等待或无效沟通。
- **证据覆盖**：是否跨多个任务、项目阶段或时间段出现，而非单次偶然事件。
- **持续时间**：问题存在了多久，以及是否长期未得到人的决策。
- **闭环必要性**：现在是否已经有足够证据和可行实验，值得占用人的注意力。

#### 慢性问题上浮机制

- 某项即便最近一两次没有进入 TOP 5，其历史触发、累计影响和绕路成本仍继续保留。
- 对长期持续的问题，近期信号可以衰减，但累计负担不能归零；持续时间会提高其被复核的优先级。
- 当累计负担或长期持续性达到值得人工决策的程度时，它应重新进入 TOP 5，并明确标记“长期累积问题”。
- 长期问题上浮时，应展示其时间跨度、累计触发次数、主要损耗和此前未进入前五的原因，避免用户误以为它是突然出现的新问题。
- 用户已经明确“不试用”或“稍后处理”的事项要保留决定历史；除非出现实质性新证据、达到用户设定的提醒日期，或累计负担明显变化，否则不能频繁重复打扰。

## 6. 决策对象与交互边界

### 6.1 候选改进

候选详情至少需要展示：

- 建议改变什么，以及它来自哪些观察。
- 预期收益、可能代价和适用范围。
- 当前证据强度与仍未知的部分。
- 三个首轮动作：稍后处理、不试用、进入试用。

### 6.2 二次确认

用户选择“进入试用”后，确认页或确认面板必须把意图变成明确计划：

- 实际会改变的内容。
- 生效范围和持续时间。
- 如何判断有效。
- 如何撤回或结束。
- 哪些部分仍需用户填写或选择。
- 建议的提醒日期；用户可以修改或取消提醒。
- 建议的成功标准；用户可以修改，并必须明确确认。

只有用户确认该计划后，才创建待 Codex 接续的交接记录；实验和实际变更仍由 Codex 在领取
交接后按知识生命周期建立。Console 不会在浏览器内调用模型，也不会直接应用变更。

实验计划应保持简短，默认只回答：

1. 要尝试什么。
2. 在什么范围尝试。
3. 如果成功，预期固化到什么载体。
4. 观察到何时或多少次任务。
5. 什么现象代表有效，什么现象代表需要停止或调整。
6. 何时提醒用户复核。

### 6.3 改进实验

实验不是单一的启用/停用开关，应保留：假设、当前采用方式、证据时间线、人工反馈、Dream 自动证据、历次调整和最终决策。

下次 Dream 应主动检查正在进行的实验是否获得了新的可用证据，并将需要人判断的实验复核纳入最多 5 项的待处理预算。

### 6.4 固化与作用范围

“固化”不是单一步骤。系统必须先展示将要改变的载体、差异和作用范围，再由用户确认：

- **项目级**：只影响当前项目或项目对应的 Dream workspace 知识与项目指引。
- **环境级**：可能影响用户在多个项目中的通用知识、全局指引或 Skill 行为。

环境级变更的影响更广，不能因实验成功而自动提升作用域；用户必须显式选择并确认。实际写入任何知识或配置载体前，还需要展示可审查的具体差异。

### 6.5 改进事项生命周期

所有候选、实验和已决定沉淀的内容统一称为“改进事项”。“改进追踪”提供完整列表，每项只能处于一个主阶段：

1. **候选**：Dream 已推荐，等待用户决定。
2. **计划中**：用户愿意试用，正在确认范围、标准与提醒。
3. **实验中**：已经确认试用计划，回到 Codex 的真实任务中积累证据。
4. **待复核**：获得足够新证据、提醒到期或长期累积负担达到阈值，需要人的判断。
5. **待落实**：用户已确认实验结论和固化方案，但尚未写入目标载体。
6. **落实中**：具体变更正在写入或等待验证，必须显示目标载体、作用范围与执行状态。
7. **已完成**：变更已写入用户确认的载体，并完成必要验证；仅“用户同意”不能算完成。
8. **已结束**：用户决定不试用、实验被终止或结论不成立；保留理由和历史证据。

允许的主要路径：

```text
候选 -> 计划中 -> 实验中 -> 待复核 -> 待落实 -> 落实中 -> 已完成
  \          \         \          \
   -> 已结束  -> 已结束  -> 实验中   -> 已结束
                         或调整后继续
```

“改进追踪”总览只展示理解状态所必需的信息：事项名称、当前阶段、作用范围、当前进度、下一步、最近变化和最终落实状态。用户点进单项后，才查看证据、决策历史、计划版本和具体载体差异。

每个单项详情必须有一条按时间排列的生命周期旅程，明确显示：由哪次 Dream 发现、用户作过什么决定、何时开始实验、获得了哪些关键证据、确认了什么固化范围、实际写入了什么，以及如何验证完成。

## 7. 视觉与交互探索约束

- 首轮只设计桌面端，基准画布 1440px。
- 已选择方案 B“聚焦工作台”作为整体视觉和交互方向。
- 使用冷中性色画布、白色工作表面、克制的钴蓝主色和少量琥珀提醒色。
- 待处理事项采用收件箱式结构，但保持个人使用尺度，不演变成运营后台。
- 使用现代中文无衬线字体，避免当前版本偏纸张、档案或复古阅读器的视觉气质。
- 强调阅读秩序与决策安全，不追求炫技或高密度运营后台感。
- 状态不能只靠颜色表达；按钮文案应描述结果而非抽象流程。
- 关键决策使用渐进披露：列表先解释“为什么值得处理”，详情再呈现证据与完整影响。

## 8. 仍需产品所有者确认

以下问题不会在设计或开发中被默认替用户决定：

1. 提醒到期后是否需要 Codex 主动唤醒用户；当前版本只在 Console 和下一次 Codex 接续时展示。
2. 各类环境级载体在真正写入前需要怎样的差异审查体验；当前版本只记录用户预期，实际写入仍在 Codex 中二次审查。

## 9. 典型用户故事

这些故事用于确认产品用途和交互闭环，不自动等同于已经批准的开发范围。

### 故事 A：从一次 Dream 发现值得试用的工作方式

**作为**一个经常让 Codex 执行复杂任务的用户，**我希望** Dream 能把反复出现的返工模式整理成一个可理解的候选改进，**从而**由我判断它是否值得试用，而不是由系统直接改变我的工作方式。

1. 用户在 Codex 中说“开始做梦”，确认复盘最近 30 天的任务。
2. Dream 发现多个任务都在范围尚未稳定时过早进入执行，并形成候选“为复杂任务增加阶段性确认”。
3. 首页“待我处理”说明候选来自哪些任务、可能改善什么、可能增加什么成本。
4. 用户点击“查看并决定”，阅读证据后选择“进入试用”。
5. 系统生成一份试用计划：只在复杂任务范围变化时触发确认，先观察 5 次任务；此时尚未应用。
6. 用户确认计划后，Console 记录人的决定并显示“等待 Codex 接续”。
7. 用户回到 Codex；Codex 领取交接、建立采用与验证记录后，实验才真正开始。

**结果：**候选从观察变成了有范围、有成功信号、可撤回的实验，而不是一条立即生效的规则。

### 故事 B：用户认为建议不适合自己的用途

**作为**拥有明确个人工作偏好的用户，**我希望**可以拒绝一个看似合理但不适合我的建议，并说明原因，**从而**避免系统反复推动同一种改进。

1. Dream 建议“每个任务都先输出完整计划”。
2. 用户打开详情，看到它可能提高可控性，但会拖慢简单任务。
3. 用户选择“不试用”，并补充原因：“只适合高风险或多阶段任务，不适合所有任务。”
4. 系统保存这次人的决定，不修改现有知识，也不把候选伪装成已实施。
5. 未来如果出现更窄的新建议，例如“只在高风险任务先输出计划”，系统可以形成新的候选，但必须同时展示此前的拒绝理由。

**结果：**拒绝不是删除记录，而是形成未来判断所需的上下文。

### 故事 C：实验有了新证据，但结果并不完全正向

**作为**正在试用一种新工作方式的用户，**我希望**看到正反证据和发生这些结果的上下文，**从而**决定继续、调整还是结束，而不是只看一个成功率。

1. “分批复核策略”已经在 4 个任务中被观察。
2. 新一次 Dream 发现其中 3 次减少了返工，但 1 次因为频繁停顿打断了连续思考。
3. 首页出现“实验复核”事项，说明为什么现在需要用户关注。
4. 用户进入实验详情，查看按时间排列的自动证据和自己过去的反馈。
5. 用户选择“调整”，把触发条件从“每完成一步”改成“每完成一个阶段”。
6. 系统展示变更前后差异；用户确认后，实验进入新版本，旧证据仍保留在原版本下。

**结果：**实验可以迭代，而不是只能简单启用或关闭；证据不会因调整而丢失。

### 故事 D：一次 Dream 没有产生新的行动项

**作为**定期复盘的用户，**我希望**即使这次没有新发现，也能确认 Dream 确实检查过正确范围，**从而**相信“没有事项”不是系统失效。

1. 用户确认复盘最近 7 天的 4 个任务。
2. Dream 完成后没有发现新的稳定模式、知识变化或候选改进。
3. 首页显示“本次没有需要你处理的新事项”，同时展示扫描范围、完成时间和完整保存状态。
4. 用户仍可打开本次 Dream，查看哪些任务被纳入以及为何没有形成候选。

**结果：**空结果是可解释的有效结果，不会出现空白页面或虚构建议。

### 故事 E：Dream 中途失败，用户稍后恢复

**作为**不希望复盘结果或进度悄悄丢失的用户，**我希望**在 Dream 未完整完成时看到明确状态和安全的恢复方式，**从而**不会误把部分结论当成完整结果。

1. Dream 已完成任务清单和部分语义复核，但在持久化知识变化前中断。
2. 首页显示“本次 Dream 未完整完成”，并说明已保存到哪个阶段、哪些内容尚未生效。
3. 未完整结果不会生成可决策候选，也不会推进复核游标。
4. 用户选择“继续本次 Dream”后，系统从安全检查点恢复；若范围发生变化，则再次请求确认。
5. 只有完整结果持久化后，才进入最近 Dream 和待处理队列。

**结果：**系统宁可显式停住，也不会让部分结果冒充完整结论。
