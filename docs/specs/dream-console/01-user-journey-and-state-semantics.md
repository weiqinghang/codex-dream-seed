# Dream Console 用户旅程与状态语义 Spec

状态：`active`
适用范围：Dream Console 基础用户可用性升级

## 目标

让不了解 Dream 内部数据模型的用户，也能从空 Workspace 开始，独立完成一次完整闭环：

`首次启动 → 开始做梦 → 查看结果 → 作出决定 → Codex 接续 → 验证 → 收尾`

Console 是确定性的查看、决策和追踪界面；Codex 是语义审阅与执行主体；Workspace 是两者共享的事实源。

## 用户旅程

### J-01 首次进入

- 空 Workspace 显示三步启动卡片，而不是只有“暂无数据”。
- 用户能理解 Dream、Console、Codex 和 Workspace 的分工。
- CTA 明确告诉用户回到 Codex 发送“开始做梦”。
- 首次使用不采用强制全屏 onboarding。

### J-02 查看一次梦境

- 梦境列表按发生时间降序。
- 左侧序号表示“第几次做梦”，最新一次显示最大的梦境序号，而不是当前渲染位置。
- 用户能看到日期、标题、状态、耗时、Token 口径和报告入口。
- 没有可靠 Token 数据时显示“未记录”，不得推算。

### J-03 理解推进看板

看板保留以下列：

1. `待决策`：Inbox，不设置 WIP 上限。
2. `试用落实`：Codex 正在落实已确认试用，但验证条件可能尚未完整建立。
3. `验证中`：试用和验证合同已经建立，正在积累正反证据。
4. `待收尾`：已达到复核窗口，需要人工判断固化、失败或证据不足。
5. `完成`：已形成可追溯终态。

不得恢复“做梦中”和“待接续”两列。`handoff_pending` 是接续机制状态，不是看板泳道。

### J-04 对待决策事项排序和决策

- 支持按价值/影响面降序。
- 支持按最早提出时间升序或降序。
- 支持按做梦提及次数降序。
- 用户可以暂缓、拒绝或制定试用计划，并在操作前看到结果和可逆性。

### J-05 理解验证

- 验证中支持按进度和已收到证据数量排序。
- 证据总数用于表示“信息充分度”；正向证据不能替代负向证据。
- 正向和负向证据必须使用可区分但可访问的颜色/图例显示。
- 详情说明哪些成功标准未满足、哪些风险影响验收、下一步建议是什么。
- 负向证据必须保留，因为它可证伪方案、触发调整或支持失败收尾。

### J-06 收尾

- 用户可以选择：固化成功、验证失败、证据不足继续观察或调整验证合同。
- 每个选择显示影响范围、证据依据和下一步。
- 完成条件不能只依赖进度百分比，必须核对成功标准与关键负向证据。

### J-07 Console 与 Codex 接续

- 用户从 Console 复制稳定定位指令到新的 Codex Session。
- 新 Session 不依赖旧聊天上下文，也能定位正确 Workspace 和事项。
- Codex 的 claim/complete/fail 写回后，Console 显示真实新状态。

### J-08 异常恢复

- handoff 失败后存在正式重试路径。
- 页面数据陈旧、API 部分失败、服务停止或 Workspace 不匹配时，用户能理解原因并恢复。
- 已成功写入但刷新失败时，不得诱导用户重复提交。

## 首页边界

首页在本轮只实现：首次空状态、帮助入口、刷新/陈旧状态和完成上述闭环所需的最小调整。

此前“WIP 超限时把首页改成收尾行动中心”只有方案，没有后续明确确认；除非仓库存在更晚的可追溯确认，不实施整体首页信息架构重做。

## 验收

- J-01 至 J-08 均有自动化或真实浏览器证据。
- 页面、详情、手册和 Agent 使用相同的状态名称与离开条件。
- 空状态、错误状态和窄屏状态不会隐藏主要 CTA。
- 已有排序、看板列和梦境降序行为不回退。
